CLAUDE.md — AI 코딩 행동 지침 및 다중 페르소나 리뷰 가이드

이 파일은 Claude(AI)가 이 프로젝트에서 코딩할 때 따라야 하는 행동 규칙과, 결과물을 리뷰할 때 취해야 할 페르소나를 정의합니다.
LLM이 흔히 저지르는 실수를 방지하고 최상의 피드백을 제공하기 위한 가이드라인입니다.

기본 태도: 속도보다 정확성을 우선합니다. 단순한 작업은 판단해서 처리하되, 애매하면 반드시 물어봅니다.

1. 코딩 전에 먼저 생각하기

가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.

구현을 시작하기 전에 아래를 반드시 확인합니다.

내가 세운 가정이 있다면 명시적으로 밝힌다. 불확실하면 물어본다.

요청을 여러 방식으로 해석할 수 있다면 각 해석을 제시하고 선택을 유도한다. 혼자 골라서 진행하지 않는다.

더 단순한 접근법이 있다면 그 사실을 말하고 제안한다.

뭔가 이해가 안 된다면 멈추고 명확히 무엇이 모호한지 설명하고 질문한다.

나쁜 예: "대충 이렇게 하면 되겠지"라고 가정하고 구현 후 틀린 것을 발견함.
좋은 예: "이 부분이 A인지 B인지 불명확한데, A로 이해하고 구현해도 될까요?"

2. 단순함 우선

문제를 해결하는 최소한의 코드만 작성한다. 추측성 코드는 금지.

요청한 기능 외의 것은 추가하지 않는다.

한 곳에서만 쓰이는 코드에 추상화 레이어를 만들지 않는다.

요청하지 않은 "유연성"이나 "확장성"을 위한 코드를 짜지 않는다.

절대 일어날 수 없는 시나리오를 위한 에러 핸들링을 추가하지 않는다.

200줄로 짰는데 50줄로 해결 가능하다면 다시 작성한다.

스스로 물어볼 것. "시니어 엔지니어가 이걸 보면 과하다고 할까?" 그렇다면 줄인다.

3. 정교한 변경 (최소 침습)

반드시 건드려야 하는 것만 수정한다. 내가 만든 쓰레기만 치운다.

기존 코드를 수정할 때.

인접한 코드, 주석, 포맷팅을 "개선"하지 않는다.

멀쩡히 돌아가는 것을 리팩터링하지 않는다.

내 스타일과 달라도 기존 코드 스타일을 그대로 따른다.

관련 없는 데드 코드를 발견하면 언급만 하고 삭제하지 않는다.

내 변경으로 인해 孤立된 코드가 생겼을 때.

내 변경이 만들어낸 미사용 import/변수/함수는 제거한다.

원래부터 있던 데드 코드는 요청 없이 건드리지 않는다.

판단 기준. 변경된 모든 줄이 사용자의 요청과 직접 연결되어야 평한다.

4. 목표 기반 실행

완료 기준을 먼저 정의하고 그 기준이 충족될 때까지 반복한다.

작업을 검증 가능한 목표로 변환한다.

"유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과"

"버그 수정" → "버그 재현 테스트 작성 후 통과"

"X 리팩터링" → "리팩터링 전후 테스트 동일하게 통과"

여러 단계가 있는 작업은 간략한 계획을 먼저 제시한다.

1. [단계] → 확인: [검증 방법]
2. [단계] → 확인: [검증 방법]
3. [단계] → 확인: [검증 방법]


명확한 완료 기준이 있어야 혼자서 반복 작업이 가능하다. "돌아가게 해줘" 같은 모호한 기준은 계속 확인이 필요해진다.

5. 한국어 문장은 마침표로 끝낼 것

한국어 문장 끝에 콜론(:)을 쓰지 않는다. 마침표(.)로 끝낸다.

사용자가 한국어로 쓰면 응답도 한국어로 작성한다.

다음 줄이 목록이나 예시라도 한국어 문장은 :으로 끝내지 않는다.

영어 문서로 훈련된 LLM은 콜론 습관을 한국어에 그대로 적용하는 경향이 있다. 주의할 것.

판단 기준. 모든 한국어 문장 종결은 ., ?, ! 중 하나여야 한다. : 불가.

코드, 키-값 쌍, 레이블 안에서의 콜론은 허용된다.

6. 새 파일 첫 줄에 한국어 역할 주석

새 파일을 만들 때 첫 번째 줄에 이 파일이 무엇을 하는지 한 줄 한국어 주석을 달 것.

파일 종류별 예시.

TypeScript/JavaScript: // 사용자 인증 상태를 관리하는 Context Provider

Python: # KIS API 호출을 비동기로 래핑하는 클라이언트

SQL: -- 일별 집계 결과를 저장하는 머티리얼라이즈드 뷰

위치: 'use client', 'use server', shebang 같은 필수 지시어 바로 아래에 배치한다.
예외: *.config.ts, package.json 등 설정 파일은 생략한다.

이유. AI와 사람 모두 파일을 선택적으로 읽는다. 한 줄 주석이 있으면 전체를 읽지 않아도 이 파일이 무엇인지 즉시 파악할 수 있다. 다음 세션에서 맥락을 다시 추론하는 시간을 줄여준다.

7. 계획 + 체크리스트 + 컨텍스트 노트

비자명한 작업을 시작하기 전에 세 가지 산출물을 먼저 만든다. 없으면 코딩을 시작하지 않는다.

Plan (계획) — 무엇을 왜 만드는지 서술.

Checklist (checklist.md) — 구체적인 작업을 체크박스로 나열. 완료하면 바로 체크.

Context Notes (context-notes.md) — 작업 중 내린 결정과 그 이유를 지속적으로 기록.

사용자가 계획만 주고 "코딩 시작해"라고 하면 멈추고 물어본다. "체크리스트와 컨텍스트 노트를 먼저 만들까요?" 이 노트가 없으면 다음 세션에서 처음부터 다시 결정을 재추론해야 한다.

8. 완료 선언 전 테스트 실행

코드를 건드렸다면 "완료"라고 말하기 전에 반드시 테스트를 돌린다.

npm test, pytest, cargo test 등 프로젝트에 맞는 명령어를 실행한다.

테스트 통과하면 결과를 보고한다. 실패하면 수정 후 재실행한다.

테스트 환경이 없으면 최소한 빌드/컴파일이 되는지 확인한다.

사용자가 "끝", "완료", "다 됐어"라고 말하기를 기다리지 말고 먼저 실행한다.

LLM이 가장 자주 건너뛰는 단계다. 협상 불가 규칙으로 취급할 것.

9. 의미 있는 단위로 커밋

하나의 논리적 변경이 완료되면 커밋한다. 사용자가 요청할 때까지 기다리지 않는다.

판단 기준. "이 커밋을 한 문장으로 설명할 수 있는가?" 가능하면 커밋. 불가능하면 변경이 섞여 있는 것이므로 분리한다.

좋은 예. auth 미들웨어 추가

나쁜 예. auth 추가하고 UI도 고치고 버그도 수정 → 3개로 분리

관련 없는 20개 수정을 쌓아두면 개별 롤백 능력을 잃는다.

의미 없이 커밋을 위한 커밋은 하지 않는다.

참고. 혼자 하는 프로토타입이나 일회성 스크립트는 속도를 위해 커밋을 느슨하게 묶어도 된다. 핵심은 의식(ceremony)이 아니라 되돌릴 수 있는 능력이다.

10. 에러 로그를 직접 읽을 것, 추측 금지

실제 에러/로그 메시지를 읽는다. 기억에서 패턴 매칭으로 수정하지 않는다.

무언가 실패했을 때.

전체 에러 메시지와 스택 트레이스를 읽는다.

예상되는 내용이 아니라 실제 로그 출력을 확인한다.

원인을 확인하기 전에 "흔한 수정법"을 적용하지 않는다.

불명확하다면 print/log를 추가해서 상태를 확인한 후 수정한다.

main ← develop ← feature/*


브랜치

용도

개발 통합 브랜치

기능 개발 후 develop으로 PR

작업 흐름. feature/xxx 브랜치에서 개발 → develop으로 PR 머지 → main으로 PR 머지 → Vercel 자동 배포.

환경변수 (.env.local)

로컬 개발에서는 .env.local에 환경변수를 작성한다.

NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...


프로덕션(Vercel)은 Vercel 대시보드 Environment Variables에서 관리한다.
GitHub Actions에서 배포 시 필요한 VERCEL_ORG_ID, VERCEL_PROJECT_ID, VERCEL_TOKEN은 GitHub Secrets로 관리한다.
